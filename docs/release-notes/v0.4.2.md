# v0.4.2 Release Notes

## Overview

Version 0.4.2 introduces an internal PKI service (step-ca) deployed under GitOps.

This release does **not** change the active certificate issuer.  
Issuer cutover to ACME is intentionally deferred.

------------------------------------------------------------------------

## Added

### Internal PKI Service (step-ca)

- step-ca deployed via providers ApplicationSet
- GitOps-managed under `gitops/manifests/step-ca`
- ACME-capable configuration
- ClusterIP service (443 â†’ 8443)
- Static PersistentVolume with `Retain` policy
- Explicit secret boundary (not GitOps-managed)

Required Kubernetes Secret:

    step-ca-secrets

Required keys:

- password
- provisioner_password

Default data path (static PV):

    /var/lib/step-ca

------------------------------------------------------------------------

## Operational Notes

- step-ca is fully operational and reachable in-cluster.
- ACME directory endpoint is active.
- No cert-manager integration is performed in this release.
- Existing internal issuer (`ClusterIssuer/int-ca`) remains authoritative.
- No ingress changes are introduced.

------------------------------------------------------------------------

## Boundary Discipline

This release intentionally separates:

1. Deployment of the PKI service
2. Issuer migration and certificate cutover

This preserves rollback safety and minimizes blast radius.

------------------------------------------------------------------------

## Upgrade Impact

- No certificate reissuance
- No DNS changes
- No ingress changes
- No secret rotation required (beyond initial step-ca secret creation)

------------------------------------------------------------------------

## Next Planned Evolution

- ACME-backed ClusterIssuer pointing to step-ca
- Controlled migration of internal ingress certificates
